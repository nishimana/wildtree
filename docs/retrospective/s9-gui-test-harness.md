## 得た知見

### 1. architect がスタブを超えて実装まで行ったプロセス逸脱

dev-cycle テンプレートでは architect の役割は「設計書 + スタブ」までで、実装は implementer が担当する。今回 architect が設計からスモークテストまで全て実装した。

結果的に実装は正しく、テストも全パスしたが、テンプレートの意図（分業による品質保証）からは逸脱している。

**分析**: S9 の特殊性として以下がある:
- テストハーネスは計測・デバッグ用の補助ツールであり、アプリケーション本体の機能ではない
- 設計書を書きながら動作確認する性質のコードだった（計測ツールの設計は実データでの動作確認と不可分）
- 結果的に「設計 + 実装」を一人が行ったことで整合性が保たれた

**判断**: テストハーネスのような補助ツールでは architect が実装まで行うことは合理的なケースがある。ただし、これはアプリケーション本体の機能開発に適用すべきではない。テンプレートのルールを変更するほどの汎用的パターンではない。

### 2. offscreen モードでのパフォーマンス問題再現の限界

実画面ではフリーズ・クラッシュする問題が、offscreen モードでは再現できなかった。スモークテスト（expand_all 含む）が全パスした。

これは offscreen と実画面で以下が異なるためと推測される:
- 実際のピクセルレンダリングが行われない
- ウィンドウマネージャとの通信がない
- GPU アクセラレーションが関与しない

**意味**: テストハーネスとスモークテストは「コードパスの正当性検証」には有効だが、「実画面でのパフォーマンス計測」には不十分。ボトルネック特定には別のアプローチ（実画面での手動計測、Qt Profiler、cProfile with GUI）が必要。

### 3. _measure ヘルパーの defensive coding

設計書では `QApplication.processEvents()` を直接呼ぶ擬似コードだったが、実装では `QApplication.instance()` の None チェックを追加した。これにより:
- QApplication が存在しないテスト環境でもクラッシュしない
- tracemalloc の start/stop も既存のトレース状態を尊重（`was_tracing` フラグ）

設計書が「最小限の擬似コード」であるのに対し、実装が「防御的に堅牢化」しているのは健全な進化。設計段階で全てのガードを書く必要はなく、実装段階での堅牢化は期待される行動。

### 4. 設計書にない追加テスト（test_select_node_and_detail）

スモークテストに設計書にない 5 つ目のテスト `test_select_node_and_detail` が追加された。ノード選択 → 詳細ペイン更新のエンドツーエンドフローを検証するテストで、設計の 4 テストを補完する妥当な追加。

architect が実装まで行ったことで、実データでの動作確認中に「これも確認すべき」と気づけた好例。ただし、通常の分業フローでは implementer がテストを追加する権限を持たないため、設計段階でカバーすべきだった。

## 設計と実装のギャップ

| ギャップ | 内容 | 対応 |
|---------|------|------|
| _measure の defensive coding | 設計: processEvents() 直接呼び出し / 実装: instance() の None チェック + tracemalloc 状態保持 | 実装が正しい。防御的コーディングは実装段階での堅牢化 |
| 設計書の未使用変数 | 設計: `current, peak = tracemalloc.get_traced_memory()` / 実装: 不要な行を正しく削除 | 実装が正しい。設計書の擬似コードのクリーンアップ |
| window プロパティの型ヒント | 設計: 型ヒントあり / 実装: なし（循環 import 回避） | 実装が正しい。循環 import 回避は設計段階で考慮すべきだった |

## 改善点

### 今後のサイクルへの適用

1. **実画面パフォーマンス問題の調査方法を確立する**: offscreen テストでは再現できない GUI パフォーマンス問題に対して、実画面での計測手法（パフォーマンスログの確認、Qt の内部プロファイリング、段階的な操作での手動計測）を整理する。テストハーネスの `[perf]` ログが実画面起動時にコンソールに出力されるので、まずはそれを確認する

2. **補助ツールの開発フローを明確化する**: テストハーネスのような補助ツールは、アプリケーション本体と異なる開発フロー（architect が実装まで行う）が合理的なケースがある。ただし、この判断は明示的に行い、なし崩し的にスコープを拡大しない

## 設計方針の修正要否

- 現時点では修正不要
- 「例外を投げない。全てのエラーを OperationResult で返す」方針は正しく機能した
- 「同じコードパスを通る」設計原則により、テストの信頼性が確保されている
- 設計 → テスト → 実装の TDD サイクルは9回連続で初回全パス（S1 除く）を達成

## 未解決の課題

### 実画面でのパフォーマンス問題

CLI 計測では問題なし、offscreen テストでも全パス。しかし実画面では依然としてフリーズ・クラッシュの可能性がある。次のステップ:
1. `python main.py [cards_dir]` で実際に起動し、コンソールの `[perf]` ログを確認
2. トップツリー「メイン」選択時の各ステップの所要時間を記録
3. expand 操作でどのステップがボトルネックか特定
4. 問題が再現する場合、Qt の `QElapsedTimer` やシステムモニタでリソース消費を確認
