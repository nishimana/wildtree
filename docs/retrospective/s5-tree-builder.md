## 得た知見

### 1. S4 振り返りの注意事項が S5 設計を正確にブートストラップした

S4 振り返りで「S5 では再帰的なロジックが入るため、S2 レベルの注意が必要」「循環参照のバリエーション、未解決参照とリテラルの混在、深いネストに注意」と具体的に記録していた。S5 設計では、これらの注意事項を設計書のエッジケーステーブル（13パターン）として展開し、テスターが41テストに変換した。

振り返りの「次に取り組むべき機能」セクションの注意事項が、2サイクル連続で次の設計を効果的にブートストラップしている。

### 2. 実データ調査が設計判断の根拠を提供した

設計前に実データの規模を調査した（最大深度9, メイン全展開3,476ノード, 循環参照142キー）。これにより:
- **パスごとの visited**: ダイヤモンド参照の存在を確認し、グローバル visited を却下できた
- **MAX_DEPTH=50**: 実データ最大9に対して十分なマージン
- **全展開**: 3,476ノードはメモリ・パフォーマンスとも問題にならない
- **パスごとの visited のコピーコスト**: 最大10要素のsetコピーは無視できる

数値に基づく判断により、設計上の曖昧さがほぼゼロになった。

### 3. 「例外を投げない設計」が実装をシンプルにした

build_tree / build_forest は例外を投げない設計とした。resolve() が None を返す場合は UNRESOLVED ノード、深さ制限超過は CIRCULAR ノードとして処理する。これにより:
- try/except ブロックが不要
- 全ての入力に対して正常な TreeNode を返す保証
- テストで例外系のケースを書く必要がない（全て正常フローとしてテスト可能）

### 4. 重複ロジックの早期抽出がレビュー指摘を軽微にした

レビューで指摘された「通常参照と動的参照 inner_ref の重複ロジック」は、`_resolve_ref()` ヘルパーへの抽出で解決した。コード量の削減だけでなく、resolve → 循環チェック → 深さチェック → REF ノード生成の判定フローが一箇所にまとまり、将来の変更が安全になった。

### 5. 設計書のノード生成判定フロー図が実装を直接ガイドした

設計書に ValueEntry からノードを生成する判定フロー（RefType.NORMAL → resolve → 循環 / 深さ / 正常 / 失敗、RefType.DYNAMIC → inner_refs 展開、リテラル → EMPTY / LITERAL）を図示したことで、`_process_value_entry()` の実装がほぼ設計書の転写になった。初回テスト全パスの要因の一つ。

### 6. 4サイクル連続で初回テスト全パス

S2 以降、S3、S4、S5 と4サイクル連続で初回テスト全パスとなった。特に S5 は再帰ロジック・循環参照検出・複数ノードタイプの判定を含む最も複雑なモジュールだったが、設計書のエッジケーステーブル + 判定フロー図 + 実データ調査の組み合わせが有効だった。

## 改善点

### 今後のサイクルへの適用

1. **S6 ではコア層（S1-S5）と GUI 層の境界に注意**: S6 は初めて PySide6 が登場する。コア層のデータモデル（TreeNode）を GUI コンポーネント（QTreeView / QStandardItemModel）にマッピングする部分で、S2 レベルの注意が必要

2. **実データ調査のパターンを維持**: S5 での実データ調査（規模・構造の事前把握）が設計判断を支えた。S6 でも実データでの表示パフォーマンスを事前検証すべき

3. **統合テストのヘルパー `_build_from_cards_dir` の重複**: 7つの統合テストクラスで同一のヘルパーが重複している（レビューの提案1で指摘済み）。テストファイルは変更不可のためスキップしたが、次のテスト作成時に conftest.py への集約を検討する

## 設計方針の修正要否

- 現時点では修正不要
- S5 の成果物（build_tree, build_forest）は S6（GUI ツリー表示）で直接使用される
- S6 設計時に確認すべき点:
  - TreeNode のプロパティ（display_name, node_type, key_def, value_entry, ref, children）が QTreeView の表示要件を満たすか
  - ノードの展開・選択・検索の UI 操作に必要な情報が TreeNode に含まれているか
  - 大きなツリー（3,476ノード）の表示パフォーマンス

## 次に取り組むべき機能

**S6: GUI -- ツリー表示（読み取り専用）**

理由:
- S5（ツリー構築）の直接の利用者。build_tree() が返す TreeNode を GUI に表示する
- v2 の最初の視覚的成果物。S1-S5 のコア層の成果を目に見える形にする
- 読み取り専用なので、編集機能（S7, S8）よりも先に取り組める

S6 では以下の点に特に注意:
- **PySide6 の QTreeView + QStandardItemModel への TreeNode マッピング**: NodeType ごとのアイコン・色分け、is_commented ノードのグレーアウト表示
- **トップツリーの一覧表示**: find_top_trees() の結果をリストペインに表示し、選択時にツリーペインを更新
- **大規模ツリーの表示パフォーマンス**: メイン（3,476ノード）の全展開表示が許容範囲内か
- **右ペインの詳細表示**: ノード選択時に KeyDefinition の情報（ファイルパス、行番号、値行一覧）を表示
- **検索機能との統合**: W3 の検索機能を v2 のツリー構造に対応させる
