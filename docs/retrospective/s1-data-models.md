## 得た知見

### 1. 実データ観察が設計精度を決定的に高めた

設計書 (v2-tree-editor.md) の仕様だけでは不十分だった。実データ（メイン.yaml、CH朝田詩乃場面.yaml）を確認して初めて以下が明確になった:

- 1行に複数の参照とリテラルがカンマ区切りで混在するパターン（`__ref1__,literal,__ref2__`）
- 動的参照の実例: `__{__cards/姦キー__}{__cards/鬼キー__}__` — 1つの動的参照内に複数の内部参照
- コメント行の中にセクション区切り（`# ---- 白球少女 ----`）が存在し、これは ValueEntry にすべきでない
- ファイルレベルのコメント（キー定義全体のコメントアウト、`# SSNN0004:` ブロック）と値行のコメントアウトの区別

この観察により、テストのエッジケースが充実し、後続フェーズでの手戻りリスクが大幅に下がった。

### 2. 「インターフェースを先に決める」原則が理想的に機能した

データモデル定義というタスクの性質上、「インターフェースを先に決める」原則がそのまま成果物になった。設計段階でフィールド名・型・デフォルト値・docstring を確定させ、スタブとして src/ に配置したため:

- テストは設計書 + スタブから書けた（実装を参照する必要がなかった）
- 実装は NotImplementedError の 1 箇所を 1 行で埋めるだけだった
- レビューでの指摘も軽微（docstring の「空リスト」→「空タプル」修正）

### 3. v1 からの設計改善が的確だった

- v1 の `is_leaf` / `is_circular` / `is_unresolved` ブールフラグは排他的だが暗黙的。NodeType Enum に統一したのは正しい判断
- v1 の `WildcardRef(name, raw)` ではパース後に name を再計算する必要があった。`full_path` を正とし `name` をプロパティにするのが自然
- v1 は `raw_values: list[str]` でコメント行を捨てていた。v2 のコメント切替機能にはコメント行の保持が必須

### 4. frozen dataclass と mutable dataclass の使い分け

- `WildcardRef` を frozen にした。`inner_refs` を tuple にする必要が生じたが、パース後に変更されないデータなので妥当。セットやディクショナリのキーとして使える利点がある
- `TreeNode` は frozen にしなかった。ツリー構築時に children を段階的に追加するパターンがあるため

## 設計と実装のギャップ

なし。設計がほぼそのまま実装に反映された。これはデータモデル定義という性質上、「設計 = インターフェース定義 = コード」の関係が成り立つタスクだったため。

## 改善点

### 今後のサイクルへの適用

1. **S2（パーサー）以降はロジック量が増える**: S1 は「設計 = コード」が成り立つ稀なケース。S2 以降はロジックの設計精度が問われる。設計書のコード例が最終版を正確に反映しているか（遭遇13回の項目）に注意
2. **conftest.py のフィクスチャ設計を早期に検討**: S2 ではテストデータとして KeyDefinition, ValueEntry, WildcardRef を多用する。S1 の型を使うテストフィクスチャを conftest.py に集約すべきか、S2 設計時に判断する
3. **実データ観察のタイミング**: 今回は設計フェーズで実データを確認したが、S2（パーサー）ではテストフェーズでも実データからテストケースを抽出すべき

## 設計方針の修正要否

- 現時点では修正不要
- S2（パーサー実装）で ValueEntry の構築が複雑になりそうな場合、`raw_text` の正規化ルールを再検討する可能性がある

## 次に取り組むべき機能

**S2: スキャナ + パーサー** (`core/scanner.py`, `core/parser.py`)

理由:
- S1（データモデル）の直接の利用者であり、依存関係上の次ステップ
- パーサーが正しく動作しないと S3 以降の全てが影響を受ける
- v1 の行ベースパーサーの知見を活かしつつ、以下を新たに対応:
  - コメント行の保持（ValueEntry.is_commented）
  - リテラル抽出（ValueEntry.literals）
  - 動的参照の構造解析（WildcardRef.inner_refs）
  - 行番号の追跡（ValueEntry.line_number, KeyDefinition.line_number）

S2 では以下の点に特に注意:
- 実データの YAML は PyYAML でパースできない特殊文字を含む（行ベースパーサーの継続採用が必要）
- Windows 環境での CRLF 処理（`splitlines()` を使う。ナレッジに既存の知見あり）
- 43,000 キー、271 ファイルでのパフォーマンス
