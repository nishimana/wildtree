## バグの根本原因: なぜこのバグが生まれたか

### 直接原因
`_rebuild_tree()` が無条件で `build_full_path_index()` を呼んでいた。これは 43,000 キーのレジストリ全体を走査してフルパスインデックスを構築する処理で、約 8.7 秒かかっていた。

### 構造的原因: 保守的すぎる安全策

S8（GUI 編集機能）の実装時、`_rebuild_tree()` は「レジストリが変更された場合に安全にツリーを再構築する」目的で設計された。将来のキー名変更やファイル追加/削除を見据えて、保守的にフルパスインデックスを再構築していた。

この判断は設計時には合理的だった。しかし、現在の唯一の呼び出し元（`_on_item_changed()` のコメント切替）ではキー名・ファイルパスが不変であり、フル再構築は論理的に不要だった。

### なぜ S8 の設計段階で気づけなかったか

1. **パフォーマンスの見積もり不足**: S8 設計書の「FullPathIndex の再構築タイミング」セクションで「全ツリー再構築方式」を採用した際、`build_full_path_index()` の実行時間（~8.7 秒）を見積もっていなかった。S6 の初回ロード時にはこの処理は問題なかった（1回だけ実行されるため）が、チェック操作のたびに呼ばれると致命的になった

2. **offscreen テストの限界**: S9 の振り返りでも記録した通り、offscreen テストではレンダリングパフォーマンスの問題を再現できない。S8 のテスト（37件）は全てパスしており、offscreen でのコードパス検証では遅延を検出できなかった

3. **実データでの手動テストが S8 完了後に行われた**: S8 の開発サイクル内では実データテストが省略され、S9 のテストハーネス整備後に初めて実データで計測した。その結果、S9 で問題が顕在化した

### 隠れた問題: stale KeyDefinition

原因特定時に発見された追加の問題として、`refresh_registry()` 後のオブジェクト参照の stale 化があった。

- `refresh_registry()` は対象ファイルの `KeyDefinition` を新しいオブジェクトに差し替える
- `full_path_index` 内の古い `KeyDefinition` 参照が stale になる
- `top_info.key_def` も stale になる

フル再構築はこの stale 問題を「副作用として」解決していた。差分更新に切り替える際、この副作用を明示的に処理する必要があった（`refresh_full_path_index()` による差分更新 + `_rebuild_tree()` でのレジストリからの最新 key_def 取得）。

## 再発防止策

### 1. 大規模データでの操作コスト見積もり

GUI イベントハンドラ（ボタンクリック、チェックボックス操作等）から呼ばれる処理パイプラインでは、各ステップの実行時間を設計段階で見積もる。特に「初回ロード時には問題ないが、頻繁に呼ばれると問題になる」パターンに注意する。

見積もり基準:
- O(N) で N=43,000 の場合、1回あたり数秒かかる可能性がある
- ユーザー操作（チェック、クリック）から呼ばれる処理は 100ms 以内が目標
- 100ms を超える見込みの場合、差分更新や遅延実行を設計する

### 2. 再構築メソッドの責務明確化

`_rebuild_tree()` のような再構築メソッドでは、「何を再構築するか」「何を再構築しないか」を docstring に明記する。呼び出し元が事前に行うべき準備（`refresh_registry()` + `refresh_full_path_index()`）も明記する。

### 3. 実データでの手動テスト

MVP 達成後の機能追加では、実データでの手動テストをサイクル内に含める。offscreen テストだけでは検出できないパフォーマンス問題がある。

## プロセスの振り返り

### 良かった点

1. **Fix サイクルが正しく機能した**: 再現確認 → 原因特定 → 修正設計 → テスト → 実装 → レビュー の流れがスムーズだった。手戻りなし、レビュー指摘も提案レベルのみ

2. **テストハーネス（S9）が修正の検証基盤として機能した**: S9 で整備したテストハーネスのパフォーマンスログにより、再現確認と効果測定が容易だった

3. **原因特定時に隠れた問題（stale KeyDefinition）を発見できた**: 単に「フル再構築を削除する」だけでなく、フル再構築が副作用として解決していた問題を特定し、差分更新で明示的に対処した

4. **回帰テストが充実**: 15件のテスト（単体10件 + 回帰5件）で差分更新の正当性とツリーの整合性を網羅的に検証した

### 改善すべき点

1. **S8 の設計段階でパフォーマンスを見積もるべきだった**: S7 の振り返りの「次に取り組むべき機能」で「FullPathIndex の再構築タイミング」を注意点として挙げていたが、設計書でコスト見積もりまで踏み込んでいなかった

2. **S8 → S9 → Fix の3サイクルで解決**: 本来 S8 の設計段階で差分更新を検討すべきだった。S9（テストハーネス）が間に挟まったのは結果的に有用だったが、パフォーマンス問題の発見が遅れた
