# 設計書: ワイルドカード ツリービューア

## 1. 概要・目的

Dynamic Prompts 拡張のワイルドカード YAML ファイルを解析し、`__参照__` の依存関係をツリー構造で表示するスタンドアロンビューア。

ワイルドカードが複雑にネストしている（作品 → キャラ → 体格 → options → ... と多段参照）ため、メイン.yaml の `シーンまとめ` だけで50行以上のコメントアウト済みキャラがある状態。全体像の把握が困難なので、ツリー構造で可視化する。

## 2. 用語定義

| 用語 | 意味 |
|---|---|
| ワイルドカード | Dynamic Prompts 拡張の `__name__` 構文で参照されるテンプレート |
| キー | YAML ファイルのトップレベルキー（例: `朝田詩乃体格:`）。ワイルドカード参照の解決先 |
| 参照 | 値の中の `__cards/xxx__` または `__xxx__` パターン。別のキーを参照する |
| エントリポイント | ツリーのルート。SD プロンプト欄に `__cards/メイン__` と書いて呼び出す |
| cards ディレクトリ | ワイルドカード YAML ファイル群のルートディレクトリ |

## 3. 機能仕様

### スコープ（初期リリース）

- ワイルドカード YAML をスキャンし、`__xxx__` 参照の依存関係をツリー表示する
- PySide6 スタンドアロンアプリとして動作する
- リテラル値（プロンプトタグ）はツリーに表示しない
- コメントアウトされた値行（`# - __xxx__`）はスキップする

### ツリー表示

エントリポイント（例: `メイン`）を選択すると、そこから参照されるキーを再帰的に展開してツリー表示する。

```
メイン
├── デフォルト
│   ├── シネマシャドウ
│   └── バックライト
├── アングル
│   └── イラスト種別
├── options/ライティングxmdl
└── シーンまとめ
    └── 朝田詩乃シーンまとめ
        └── 朝田詩乃
            ├── 朝田詩乃タイトルプロンプト
            ├── 朝田詩乃キャラlora
            ├── 朝田詩乃体格
            │   ├── 朝田詩乃胸サイズ
            │   └── options/エイジスライダーN05
            ├── 朝田詩乃髪型
            ├── 朝田詩乃目
            └── 朝田詩乃性格ベース
```

### 操作

- ノードの展開/折り畳み（QTreeWidget 標準）
- エントリポイントの選択（トップレベルキー一覧から QComboBox で選ぶ）
- 更新ボタン（YAML を再スキャンしてツリーを再構築）

### スコープ外（将来）

- ノード選択 → 右ペインにキー定義の中身を表示
- キー定義のインライン編集（YAML を直接編集して保存）
- 壊れた参照のハイライト
- 検索

## 4. 技術設計

### 4.1 アーキテクチャ概要

```
wildtree/
├── main.py                 エントリポイント
├── core/
│   └── wildcard_parser.py  YAML スキャン + キー抽出 + 参照抽出 + 名前解決 + ツリー構築
├── gui/
│   └── main_window.py      QTreeWidget によるツリー表示
├── tests/
│   ├── test_parser.py      パーサーのユニットテスト
│   └── test_data/          テスト用 YAML ファイル
├── docs/
│   └── design/
│       └── wildcard-tree-viewer.md  (この文書)
└── requirements.txt        PySide6
```

パーサー（core）と GUI を分離する。パーサーは PySide6 に依存しない純粋な Python モジュールとし、テスト容易性を確保する。

### 4.2 ワイルドカードパーサー (`core/wildcard_parser.py`)

パーサーの処理は5段階のパイプラインになる。

#### Stage 1: YAML ファイルの再帰スキャン

cards ディレクトリを再帰走査し、`.yaml` / `.yml` ファイルの一覧を取得する。

```python
def scan_yaml_files(cards_dir: Path) -> list[Path]:
    """cards ディレクトリ配下の YAML ファイルを再帰取得"""
```

参考: `dynamic_linter` の `WildcardValidator.scanYAML()` と同等の処理。

#### Stage 2: キー定義の抽出（行ベースパーサー）

各 YAML ファイルからトップレベルキーを抽出する。インデントなし・`#` 以外の行から `key:` パターンを検出する。フル YAML パーサー（PyYAML）は使わず、行ベースで処理する。

値行（`  - ...` や `  # - ...`）のうち、`#` で始まるもの（先頭空白を除去後）はコメントアウトとしてスキップする。

```python
@dataclass
class KeyDefinition:
    name: str           # キー名 (例: "朝田詩乃体格")
    file_path: Path     # 定義元ファイル
    raw_values: list[str]  # キーに紐づく値行のリスト（コメント行を除く）

def extract_keys_from_file(file_path: Path) -> list[KeyDefinition]:
    """YAML ファイルからトップレベルキーとその値を抽出"""
```

**行ベースにする理由**: Dynamic Prompts の YAML はプロンプト文字列をそのまま値に持つため、厳密な YAML パースだと特殊文字（`(`、`:`、`{}`）で失敗するケースがある。`dynamic_linter` でも行ベースパーサーを採用している。

参考: `dynamic_linter` の `WildcardValidator.extractKeysFromText()`。

#### Stage 3: 参照パターンの抽出

値行の中から `__xxx__` パターンを抽出する。ネスト（`__{__inner__}outer__`）に対応する必要がある。

```python
@dataclass
class WildcardRef:
    name: str    # 参照名 (例: "cards/ソードアート・オンライン/CH朝田詩乃/朝田詩乃体格")
    raw: str     # 元テキスト (例: "__cards/ソードアート・オンライン/CH朝田詩乃/朝田詩乃体格__")

def extract_refs_from_line(line: str) -> list[WildcardRef]:
    """行内の全 __name__ 参照を抽出（ネスト再帰対応）"""
```

参考: `dynamic_linter` の `parseWildcardSpansOnLine()` + `scanLineWildcard()`。ブレース深度を追跡して閉じ `__` を検出するロジックを Python に移植する。

#### Stage 4: 名前解決

参照名をキー定義に解決する。2種類の参照形式を処理する。

```python
class WildcardResolver:
    def __init__(self, keys: dict[str, list[KeyDefinition]], cards_dir: Path):
        """keys: キー名 → KeyDefinition リストのレジストリ（Section 5 参照）
        cards_dir: フルパス参照の解決に使うルートディレクトリ"""

    def resolve(self, ref_name: str) -> KeyDefinition | None:
        """参照名をキー定義に解決する"""
```

解決ルールの詳細は「5. 参照解決アルゴリズム」を参照。

#### Stage 5: ツリー構築

解決済みの参照を再帰的にたどってツリーを構築する。循環参照を検出して無限ループを防ぐ。

```python
@dataclass
class TreeNode:
    name: str                    # 表示名
    ref_name: str                # 参照名（フルパス）
    children: list['TreeNode']   # 子ノード
    is_leaf: bool                # 参照先が見つからない
    is_circular: bool            # 循環参照検出フラグ

def build_tree(entry_key: str, resolver: WildcardResolver) -> TreeNode:
    """エントリポイントから再帰的にツリーを構築"""
```

循環参照検出: 再帰中に訪問済みキーのセットを保持し、既に訪問済みのキーに遭遇した場合は `is_circular=True` として打ち切る。

## 5. 参照解決アルゴリズム

### 参照の2形式

| 形式 | 例 | 意味 |
|---|---|---|
| フルパス | `__cards/ソードアート・オンライン/CH朝田詩乃/朝田詩乃体格__` | `cards/` 配下のパス構造でキーを特定 |
| 短縮形 | `__朝田詩乃体格__` | キー名だけで検索（同名は後勝ち） |

### フルパス解決

`__cards/ソードアート・オンライン/CH朝田詩乃/朝田詩乃体格__` の場合:

1. `cards/` プレフィックスを除去 → `ソードアート・オンライン/CH朝田詩乃/朝田詩乃体格`
2. スラッシュで分割 → 最後の要素がキー名（`朝田詩乃体格`）、それより前がディレクトリパス
3. キー名で候補を検索し、**定義元ファイルのパスがディレクトリパスと一致する**ものを選択

同名キーが別ディレクトリに存在する場合でも、ファイルパスの照合により正しいキーに解決できる。

### 短縮形解決

`__朝田詩乃体格__` の場合:

1. 全ファイルから抽出した全キーの中から `朝田詩乃体格` を検索
2. 同名キーが複数ファイルにある場合は後勝ち（スキャン順で最後に見つかったもの）

### 名前解決のデータ構造

```python
# Stage 2 の結果から構築
# キー名 → KeyDefinition のリスト（同名キーが複数ファイルにある場合に備える）
key_registry: dict[str, list[KeyDefinition]] = defaultdict(list)

for file_path in sorted(yaml_files):  # ソートしてスキャン順を安定化
    for key_def in extract_keys_from_file(file_path):
        key_registry[key_def.name].append(key_def)
```

- フルパス参照: `key_registry[key_name]` からファイルパスが一致するものを選択
- 短縮形参照: `key_registry[key_name]` の最後の要素を選択（後勝ち）

### ネスト参照 (`__{__inner__}outer__`)

Dynamic Prompts はブレース `{}` 内で選択肢を表現する。例:

```
__{__season__}_{__character__}__
```

この場合、`__season__` と `__character__` がまず解決され、その結果を結合したキー名が最終的な参照先になる。

ツリービューアでは:
- 外側の参照は「動的参照（実行時に決まる）」としてノード表示する
- 内側の `__season__`、`__character__` は子ノードとして展開する

## 6. エッジケース・制約

### 循環参照

キー A → キー B → キー A のような循環。再帰中の訪問済みセットで検出し、循環ノードとしてマークして打ち切る。

### 同名キーの後勝ち

複数ファイルで同じキー名が定義されている場合、スキャン順で最後のものが勝つ。これは Dynamic Prompts の挙動に合わせる。スキャン順はファイルパスのソート順（`sorted(Path.rglob(...))`)で安定化する。

### コメントアウトされた値行

値行が `#` で始まる場合（先頭空白を除去後）はスキップする。これにより `シーンまとめ` 内のコメントアウト済みキャラがツリーに含まれない。

```yaml
シーンまとめ:
  # - __cards/ブルーアーカイブ/...__    ← スキップ
  - __cards/ソードアート・オンライン/...__  ← 処理する
```

### 参照先が見つからない

`__cards/存在しないキー__` のように解決できない参照。初期リリースではツリーに表示しない（リテラル値と同じ扱い）。将来の「壊れた参照ハイライト」機能で対応する。

### ファイルエンコーディング

日本語ファイル名・日本語キー名を含むため、UTF-8 でのファイル読み取りを前提とする。

### パフォーマンス

cards ディレクトリの規模感:
- 作品数: 数個〜十数個
- 作品あたりのキャラ: 数人〜十数人
- キャラあたりのファイル: 3〜10個
- 合計: 数十〜数百ファイル、数百〜数千キー

初回スキャンは起動時に同期実行で問題ない規模。
