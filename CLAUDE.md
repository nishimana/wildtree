# WildTree

Stable Diffusion Dynamic Prompts 拡張機能のワイルドカード YAML をツリー構造で表示・編集するデスクトップアプリケーション。
271個・43,000キー定義規模のワイルドカードを、エントリポイントからツリーとして正しく展開し、その場で編集できるツールを目指す。

## 起動

```bash
# 通常起動
python main.py [cards_dir]

# ダブルクリック起動（コンソールなし）
wildtree.pyw
```

## 構成

```
wildtree/
├── main.py                 # エントリポイント
├── wildtree.pyw            # ダブルクリック起動用

```

## 技術的な注意事項

- Dynamic Prompts の YAML はプロンプト文字列をそのまま値に持つため、PyYAML では特殊文字（`(`、`:`、`{}`）でパースが失敗するケースがある。行ベースパーサーを採用している
- 日本語ファイル名・日本語キー名を含むため、ファイル読み取りは UTF-8 前提
- Windows 環境でのパスのエンコーディングに注意（`python -X utf8` や `sys.stdout.reconfigure(encoding='utf-8')` が必要になる場合がある）
- 現在のコードはプロトタイプ。v2 の設計は白紙から行い、プロトタイプの実装に引っ張られないこと
- 実データの cards ディレクトリ: `C:\Users\nishi\OneDrive\デスクトップ\git\webui_forge_cu121_torch231\webui\extensions\sd-dynamic-prompts\wildcards\cards`

## 開発サイクル

このプロジェクトでは以下のサイクルで開発・保守を進める。

### 新機能の開発

```
設計 → テスト → 実装 → レビュー → 修正 → 振り返り → 次の設計
```

### バグ修正

```
再現確認 → 原因特定 → 修正設計 → テスト追加 → 修正実装 → レビュー → 振り返り
```

### 機能改善

```
現状分析 → 改善設計 → テスト更新 → 改善実装 → レビュー → 振り返り
```

### リファクタリング

```
テスト確認（ゲート） → コード分析 → リファクタ計画 → 段階的実行 → レビュー → 振り返り
```

### 各フェーズの原則

- **設計**: 機能のインターフェース（関数名、引数、戻り値の型）と振る舞い（正常系・異常系）を明確に定義する
- **分析**: 既存コードの構造・問題点・影響範囲を明確にしてから変更に着手する
- **テスト**: 設計に基づいてテストを先に書く。リファクタ時はまず既存の振る舞いを固定するテストを書く
- **実装**: テストをすべてパスするコードを書く。テストファイルは変更しない
- **レビュー**: コード品質、設計との整合性、テストの漏れを確認する。修正系では根本原因への対処を重点確認する
- **修正**: レビュー指摘に対応する。修正後にテストがすべて通ることを確認する
- **振り返り**: 今回の知見を記録し、設計方針の修正要否を判断する

### プロトタイプについて

- プロトタイプは検証目的で作る。作り込まず、確認したいことだけ確認する
- プロトタイプ後は設計エージェントに再設計を依頼してから本実装に入る
- プロトタイプのコードは本実装で捨てる前提とする

### コマンド

| コマンド | 用途 |
|---|---|
| `/dev-cycle [機能名]` | 新機能の開発サイクル |
| `/fix [バグの説明]` | バグ修正サイクル |
| `/improve [改善対象]` | 機能改善サイクル |
| `/refactor [対象]` | リファクタサイクル |
| `/prototype [機能名]` | プロトタイプ作成→分析→再設計 |
| `/wrap` | 品質ゲート＋コミット |
| `/health-check` | プロジェクト全体の一貫性チェック |

### コマンドの使い分け

| やりたいこと | コマンド |
|---|---|
| 新しい機能をゼロから作りたい | `/dev-cycle` |
| バグを直したい（正しく動かない） | `/fix` |
| 動いてるけど性能や使い勝手を良くしたい | `/improve` |
| 動いてるけどコードが汚い・構造を直したい | `/refactor` |
| 技術検証やUIの確認をしたい | `/prototype` |
| 作業の区切りでコミットしたい | `/wrap` |
| プロジェクト全体を見直したい | `/health-check` |

### 全体チェック

- 3〜5機能ごとに `/health-check` で全体の一貫性チェックを行う

## サブエージェント構成

| エージェント | 役割 | ツール権限 |
|---|---|---|
| architect | 設計・分析・振り返り | 読み取り + `docs/` への書き込み |
| tester | テストコード作成 | 読み取り + テストファイルへの書き込み（テスト実行はしない） |
| implementer | 実装・修正 | フルアクセス（テストファイル変更禁止） |
| reviewer | コードレビュー | 読み取り専用 |

## テスト

- フレームワーク: pytest
- テスト実行: `python -m pytest tests/ -v`
- ビルド: なし
- 型チェック・リンター: なし
- テストファイルは `tests/` に配置
- 設計ドキュメントは `docs/design/` に配置
- 振り返りログは `docs/retrospective/` に配置

## ナレッジベース

ナレッジは `~/.claude/knowledge/dev-cycle/` に蓄積される。

- 汎用原則: `principles.md`
- 技術固有: `python.md` / `pyside6.md` / `windows.md`

### 運用方針

- **設計・テスト・実装・レビュー時にはナレッジを読まない**。事前に「これに注意しろ」と伝えても行動は変わらない
- **振り返り時（ステップ6）にのみ読み書きする**。今回の作業で得た知見をナレッジファイルに記録する
- 蓄積されたナレッジは、指示の抽象度・手順・エージェント定義などの**仕組みの改善**に活用する

## コーディング規約

- コメントは日本語で書く

- 振り返りログは `docs/retrospective/` に配置する

## コミット規約

- 日本語でコミットメッセージ
- 種別: Feat / Fix / Refactor / Improve / Style / Docs / Chore / Perf
- 例: `Feat: ユーザー認証機能を追加`、`Fix: ログイン時のバリデーションエラーを修正`、`Improve: ユーザー検索のレスポンス速度を改善`、`Refactor: ユーザー管理モジュールの責務を分離`
