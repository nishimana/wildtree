## 得た知見

### 1. 行ベースパーサーの「暗黙の前提」は実データで壊れる

設計書では「空行はスキップ」と明記していたが、実装時に「空行でキー定義が途切れる」バグが発生した。原因は「インデントなしの空行」を新しいキーの区切りとして扱ってしまったこと。実データでは空行がキー定義の値ブロック中に頻出するため、空行を無条件にスキップ（現在のキーを閉じない）する動作が必要だった。

設計書にルールとして書いていても、実装時に行の分岐ロジック（インデントあり/なし × 空行/コメント/キー行/値行）の組み合わせを正確にカバーするのは難しい。テストが先に書かれていたおかげで初回テスト実行時にすぐ検出できた。

### 2. DP 構文のブレース内カンマは専用の分割ロジックが必要

`extract_literals` でカンマ分割を行う際、DP 構文 `{...|...}` 内のカンマも分割してしまうバグが発生した。`_split_top_level_commas` 関数にブレース深度追跡を導入して解決した。

これは設計段階で予見できた問題だった。設計書のエッジケース表に DP 構文を含むリテラルパターンを列挙していたが、「カンマ分割がこれらのパターンにどう影響するか」の分析が不足していた。エッジケースを列挙するだけでなく、各処理ステップ（参照抽出 -> カンマ分割 -> リテラル収集）がそのエッジケースに対してどう振る舞うかの「処理ステップ x エッジケース」マトリクスがあれば設計段階で検出できた。

### 3. テスト先行の効果が明確に現れた

109 テストが先に書かれていたことで、初回テスト実行時に 2 件のバグを即座に検出・修正できた。手戻りのスコープが「初回テスト → 2 件修正 → 全パス」と非常に小さく収まった。S1（手戻りゼロ）とは異なるが、ロジック量が大幅に増えた S2 で手戻り 2 件は良好な結果。

### 4. S1 振り返りの提案が有効に機能した

S1 の振り返りで挙げた 3 つの改善点がすべて S2 で活用された:
- 「conftest.py のフィクスチャ設計を早期に検討」-> 設計書にフィクスチャ設計セクションを含め、3 件の新規フィクスチャを目的別に分離して追加
- 「S2 以降はロジック量が増える」-> 実際に 2 件のバグが発生。テスト先行で対処
- 「テストフェーズでも実データからテストケースを抽出」-> エッジケース表の 17 パターンが実データベース

### 5. レビュー指摘はコードの読みやすさに集中

レビューの 4 件はすべてコード品質に関するもので、ロジックの誤りではなかった:
- コメントとロジックの乖離修正（コメントが実装の動作と合っていなかった）
- 未使用変数の削除
- エスケープクォート対応の追加（`_split_top_level_commas` でバックスラッシュエスケープを考慮）
- `lstrip("#")` の意図コメント追加（「`##` や `###` も除去するが意図的」であることの明示）

「コメントとロジックの乖離」は既存ナレッジ「設計書内のコード例・図は最終版を反映する」の派生パターンだが、今回は設計書ではなく実装コード内のコメントで発生した。

## 設計と実装のギャップ

### ギャップ 1: 空行処理の設計不足

設計書では「空行はスキップ」とだけ書いていたが、「空行はスキップ」の意味が 2 通り解釈可能だった:
- (A) 空行をスキップして次の行に進む（現在のキーは閉じない）
- (B) 空行でキー定義の値ブロックが終了する

正しいのは (A) だが、設計書が曖昧だったため実装で (B) に近い動作になった。

**教訓**: 行ベースパーサーの設計では、各行タイプ（空行、コメント行、キー行、値行）に対して「現在のパーサー状態をどう変化させるか」を状態遷移表で明示すべき。

### ギャップ 2: カンマ分割の深度考慮

設計書のリテラル抽出セクションでは「カンマ区切りの値行を分割」とだけ書いていた。実際には DP 構文 `{...}` 内のカンマは分割してはいけない。

**教訓**: 「分割」「抽出」「除去」等の文字列操作を設計する際、対象テキストに含まれる構文構造（ブレース、クォート等）が操作にどう干渉するかを設計段階で分析すべき。

## 改善点

### 今後のサイクルへの適用

1. **行ベースパーサーの状態遷移を設計書で明示する**: 各行タイプ x パーサー状態のマトリクスを作り、曖昧さを排除する。S2 では「空行 x キー定義中 = 継続」が暗黙だったため実装バグに繋がった

2. **文字列操作設計時に構文干渉を分析する**: 「カンマで分割する」と設計する場合、対象テキストのどの構文がカンマを含みうるかを列挙する。今回の DP 構文 `{...|...}` は設計書のエッジケース表に存在していたが、分割ロジックとの干渉が分析されていなかった

3. **「処理ステップ x エッジケース」マトリクスの作成**: エッジケースを列挙するだけでなく、各処理ステップがそのエッジケースに対してどう振る舞うかを設計段階で確認する。これにより「あるエッジケースがステップ A では問題ないがステップ B で問題になる」パターンを事前検出できる

## 設計方針の修正要否

- 現時点では修正不要
- S3（名前解決）では、パーサーの出力（KeyDefinition, WildcardRef）を消費する。パーサーの出力形式が S3 の入力として十分かを S3 設計時に検証する

## 次に取り組むべき機能

**S3: 名前解決** (`core/resolver.py`)

理由:
- S2（パーサー + スキャナ）の直接の利用者。パーサーの出力（KeyRegistry）を使って参照を解決する
- S4（トップツリー検出）、S5（ツリー構築）の前提となる基盤機能
- v1 のリゾルバの知見を活かしつつ、以下を新たに対応:
  - 短縮形参照の解決ルール（後勝ち）
  - フルパス参照の解決
  - 循環参照の検出
  - 未解決参照の記録（壊れた参照ハイライト用）

S3 では以下の点に特に注意:
- S2 の出力形式（KeyRegistry の構造）が S3 の処理に十分かの確認
- 循環参照検出のアルゴリズム設計（visited セットのスコープ管理）
- 解決結果をどのデータ構造で返すか（v2 モデルとの整合）
