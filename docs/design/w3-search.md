# W3: 検索機能（ツリー内キー名検索）

## 機能の概要

QTreeWidget 内のノードテキストを部分一致で検索し、マッチするノード間をナビゲーションできる機能を追加する。

## タスク種別

**既存コード変更**: `gui/main_window.py` に UI 要素（検索バー）と検索ロジックを追加する。`core/wildcard_parser.py` の変更は不要。

## 設計判断の理由 (why)

### 検索対象は QTreeWidget のノードテキスト（core の TreeNode ではない）

検索は GUI レイヤーの機能であり、QTreeWidget 上のテキストを対象にする。core の TreeNode を直接検索する方法も考えられるが、以下の理由で QTreeWidget ベースにする:

- ツリー表示はエントリポイント選択のたびに再構築される。検索結果も表示中のツリーに対して有効であればよい
- QTreeWidget には `findItems()` メソッドがあり、Qt の標準機能として部分一致検索が可能。自前で再帰的走査を書く必要がない
- `(circular ref)` ラベル付きのテキストも検索対象になるため、表示テキストベースの方がユーザーの直感に合う
- core に検索機能を追加すると、core が GUI の表示状態に依存する構成になり、core/gui 分離の原則に反する

### `QTreeWidget.findItems()` の活用

`QTreeWidget.findItems(text, flags, column)` は Qt 標準の検索メソッドで、`Qt.MatchFlag.MatchContains | Qt.MatchFlag.MatchRecursive` を指定することで再帰的な部分一致検索が可能。自前の再帰走査を書く必要がなく、Qt のネイティブ実装に委ねられる。

### 検索バーをツリー上部ではなく中段バー（エントリポイント選択の下）に配置する

- ツリー左ペイン内に検索バーを入れると、スプリッタ内のレイアウトが複雑になる
- 中段バー（`mid_bar`）の下に新しい行として追加すれば、既存レイアウト構造への影響が最小限
- 検索バーはツリー操作の一部だが、ウィンドウ幅全体を使える方がテキスト入力しやすい

### ナビゲーション方式: 「次へ」「前へ」ボタン

- 全マッチをハイライトする方式も検討したが、QTreeWidgetItem の個別背景色管理は `is_unresolved` の赤字表示と競合する恐れがある
- 「現在のマッチにスクロール + 選択」方式の方がシンプルで、既存の `currentItemChanged` シグナルの仕組みをそのまま活用できる（マッチノードを選択すると右ペインにキー定義が表示される）
- マッチ件数の表示（例: "3/10"）を添えることで、全体の検索結果を把握できる

### 大文字/小文字を区別しない（case-insensitive）

- ワイルドカードキー名は日本語と英語が混在する。英語部分で大文字小文字を区別しても利便性が低い
- `Qt.MatchFlag.MatchContains` に `Qt.MatchFlag.MatchCaseSensitive` を付けなければ case-insensitive になる（Qt のデフォルト動作）

### 却下した代替案

- **core に検索メソッドを追加する案**: core/gui 分離の原則に反する。検索は表示レイヤーの機能
- **QTreeWidget 内にインライン検索を配置する案**: スプリッタ内のレイアウトが複雑化する
- **全マッチを同時にハイライトする案**: `is_unresolved` の赤字表示との競合が複雑。W3 ではナビゲーション方式に集中する
- **正規表現検索**: ユーザーにとって過剰。部分一致で十分
- **検索結果リスト（別ペイン）を表示する案**: UI が複雑化する。ツリー内での選択ナビゲーションの方が文脈を保てる

## スコープ (scope)

### やること

- 検索バー（QLineEdit + 「前へ」「次へ」ボタン + マッチ件数ラベル）の UI 追加
- QTreeWidget のノードテキストに対する部分一致検索
- マッチノード間のナビゲーション（次へ/前へでループ）
- マッチノードへの自動スクロールと選択
- 検索テキスト変更時の自動再検索
- ツリー再構築時の検索結果クリア

### やらないこと（意図的な対象外）

| 対象外 | 理由 |
|---|---|
| 右ペイン（キー定義テキスト）の検索 | W3 はツリーのキー名検索に集中する。右ペイン検索は将来の機能 |
| 正規表現検索 | ユーザーにとって過剰。部分一致で十分 |
| 全マッチの同時ハイライト | `is_unresolved` 赤字表示との競合を避ける。W3 ではナビゲーションのみ |
| キーボードショートカット（Ctrl+F等） | 将来の改善。W3 では UI ボタンのみ |
| 検索履歴 | 不要な複雑さ |
| フィルタリング（マッチしないノードを隠す） | ツリーの文脈が失われる。検索ナビゲーションの方が適切 |
| 大文字/小文字の区別オプション | デフォルトで case-insensitive。オプション追加は将来の改善 |

## 設計原則

### 検索状態の管理

implementer は以下の状態を MainWindow の属性として管理する:

1. **マッチ結果リスト** (`list[QTreeWidgetItem]`): `findItems()` の戻り値を保持する
2. **現在のインデックス** (`int`): マッチ結果リスト内の現在位置（-1 は「未選択」または「マッチなし」）

検索テキストが変更されたら、マッチ結果を再計算し、インデックスを 0（先頭）にリセットする。

### ナビゲーションのループ動作

- 「次へ」: インデックスを +1。最後のマッチの次は先頭に戻る（ラップアラウンド）
- 「前へ」: インデックスを -1。先頭のマッチの前は最後に移動する（ラップアラウンド）
- マッチが 0 件のとき、ナビゲーションボタンは何もしない

### マッチ件数の表示フォーマット

- マッチあり: `"3/10"` （現在位置 / 全件数、1-based）
- マッチなし（検索テキストあり）: `"0/0"`
- 検索テキストが空: `""` （空文字列、ラベル非表示または空）

### ツリー再構築時の検索状態クリア

`_build_and_display_tree()` でツリーが再構築されたとき、検索結果は無効になる（QTreeWidgetItem の参照が古くなる）。ツリー再構築時に:
- マッチ結果リストをクリアする
- インデックスを -1 にリセットする
- マッチ件数ラベルをクリアする
- 検索テキスト（QLineEdit）はクリアしない（ユーザーの入力を保持する）
- ただし、検索テキストが残っている場合は再検索を行い、新しいツリーに対してマッチ結果を更新する

### マッチノードの選択と右ペインの連動

ナビゲーションで `_tree_widget.setCurrentItem(match_item)` を呼ぶと、既存の `currentItemChanged` シグナルが発火し、`_on_tree_item_selected()` 経由で右ペインにキー定義が表示される。追加のシグナル接続は不要。

### マッチノードの可視性

`_tree_widget.scrollToItem(match_item)` を呼ぶことで、マッチノードが折りたたまれたサブツリー内にあっても自動展開される。ただし `scrollToItem` は親が展開されていないと正しくスクロールしない場合がある。安全のため、マッチノードの全祖先を `setExpanded(True)` で展開してから `scrollToItem` を呼ぶ。

### docstring の更新（W2 振り返りの教訓）

W2 の振り返りで「ロジック変更時の docstring 更新を変更指示に含める」が教訓として記録された。W3 では以下に注意する:
- `_setup_ui()` の docstring: 検索バーの記述を追加する
- `_connect_signals()` の docstring: 検索関連のシグナル接続を追加する
- `_build_and_display_tree()` の docstring: 検索状態クリアの記述を追加する
- `MainWindow` クラスの docstring: レイアウト図に検索バーを追加し、Attributes に新規属性を追加する

### 防御的コード・到達困難コードのコメント（W2 振り返りの教訓）

W2 の振り返りで「到達困難なコードパスにはコメントを付ける」が教訓として記録された。W3 で防御的コードを書く場合（例: マッチ結果が空なのにナビゲーションが呼ばれるケース）、そのコードにコメントで理由を記述する。

## 既存コードとの関係 (context)

### データの流れ

```
[ユーザーが検索テキストを入力]
    |
    v
QLineEdit.textChanged シグナル
    |
    v
_on_search_text_changed(text: str)
    |
    v
QTreeWidget.findItems(text, MatchContains | MatchRecursive, 0)
    |
    v
list[QTreeWidgetItem]  -- マッチ結果を保持
    |
    v
_navigate_to_match(index)
    |
    v
_tree_widget.setCurrentItem(match_item)  -- 選択
_tree_widget.scrollToItem(match_item)    -- スクロール
    |
    v
currentItemChanged シグナル（既存）
    |
    v
_on_tree_item_selected()（既存）→ 右ペインにキー定義表示
```

### 影響を受ける既存メソッド

| メソッド | 変更内容 |
|---|---|
| `_setup_ui()` | 検索バー UI の追加（中段バーの下に新しい行） |
| `_connect_signals()` | 検索関連シグナルの接続追加 |
| `_build_and_display_tree()` | ツリークリア時に検索状態もクリア |

### 変更しないメソッド

- `__init__()`: `_setup_ui()` と `_connect_signals()` の呼び出し順序は変わらない。検索状態の初期化は `_setup_ui()` 内で UI と同時に行える
- `_on_browse()`, `_on_refresh()`, `_on_entry_changed()`: 変更不要（`_build_and_display_tree()` 経由で検索状態がクリアされる）
- `_on_tree_item_selected()`: 変更不要（マッチノード選択時に既存のシグナル経由で呼ばれる）
- `_format_key_definition()`: 変更不要
- `_populate_tree_item()`: 変更不要
- `_load_cards_dir()`: 変更不要

### core モジュールとの関係

`core/wildcard_parser.py` の変更は一切不要。検索は QTreeWidget の `findItems()` を使い、core のデータモデルには触れない。

## 正常系の振る舞い

### 検索テキスト入力でマッチノードが見つかる

1. ユーザーが検索ボックスに文字列を入力する（例: "体格"）
2. `textChanged` シグナルが発火し、`_on_search_text_changed("体格")` が呼ばれる
3. `findItems("体格", MatchContains | MatchRecursive, 0)` が呼ばれる
4. マッチする QTreeWidgetItem のリストが返る（例: 3 件）
5. インデックスが 0 にセットされ、最初のマッチノードが選択・スクロールされる
6. マッチ件数ラベルに `"1/3"` と表示される
7. 右ペインに選択されたノードのキー定義が表示される（既存の `_on_tree_item_selected()` 経由）

### 「次へ」ボタンでマッチ間を移動する

1. マッチが 3 件あり、現在インデックスが 0 (1/3) の状態
2. ユーザーが「次へ」ボタンをクリックする
3. インデックスが 1 (2/3) になり、2 番目のマッチノードが選択・スクロールされる
4. もう一度「次へ」→ インデックス 2 (3/3)
5. もう一度「次へ」→ インデックス 0 (1/3) にラップアラウンド

### ツリー再構築後の再検索

1. 検索ボックスに "体格" と入力されている状態
2. ユーザーがエントリポイントを変更する
3. `_build_and_display_tree()` が呼ばれ、ツリーが再構築される
4. 検索状態がクリアされる
5. 検索テキスト "体格" が残っているので、新しいツリーに対して再検索が実行される
6. 新しいマッチ結果が表示される

## 異常系の振る舞い

| ケース | 処理 |
|---|---|
| 検索テキストがマッチしない | マッチ結果は空リスト。マッチ件数ラベルに `"0/0"`。ナビゲーションボタンは何もしない |
| 検索テキストが空 | マッチ結果は空リスト。マッチ件数ラベルは空文字列。現在のツリー選択は変更しない |
| ツリーが空（cards_dir 未読込） | `findItems()` は空リストを返す。マッチ件数ラベルに `"0/0"` |
| ツリー再構築中に検索が走る | `_build_and_display_tree()` でクリアしてから再検索するので、古い QTreeWidgetItem への参照は保持されない |

## エッジケース

### `(circular ref)` ラベル付きノードの検索

`is_circular=True` のノードは表示テキストが `"キー名 (circular ref)"` になる。"circular" で検索するとこれらのノードがマッチする。これは意図通りの動作（表示テキストベースの検索なので、ユーザーが見えるテキストがそのまま検索対象）。

### 赤字（unresolved）ノードの検索

`is_unresolved=True` のノードも通常のテキスト検索でマッチする。検索で選択した場合、右ペインには「(キー定義が見つかりません)」が表示される（既存動作のまま）。

### 1 文字ずつの入力による連続検索

`textChanged` はキー入力のたびに発火する。"体格" と入力する過程で "体" → "体格" と 2 回検索が走る。`findItems()` はインメモリのツリーに対する操作であり、パフォーマンス問題は発生しない（ツリーのノード数は通常数百程度）。

### マッチ結果内のノードが折りたたまれている場合

ナビゲーション時に祖先ノードを展開してからスクロールするため、ユーザーが折りたたんだサブツリー内のノードにもナビゲーションできる。

### 検索テキストに特殊文字が含まれる場合

`findItems()` は `MatchContains` フラグでプレーンテキスト検索を行う。正規表現ではないため、`()` や `[]` 等の特殊文字はリテラルとして扱われる。

## テストの指針

### テストパターン

**検索ロジック**:
- 検索テキスト入力でマッチノードが見つかる
- 検索テキスト入力でマッチなし → ラベルが `"0/0"`
- 空文字列の検索 → マッチなし、ラベルが空
- 部分一致で検索できる
- case-insensitive で検索できる

**ナビゲーション**:
- 「次へ」でインデックスが進む
- 「前へ」でインデックスが戻る
- 最後のマッチの「次へ」で先頭に戻る（ラップアラウンド）
- 先頭のマッチの「前へ」で最後に移動する（ラップアラウンド）
- マッチなしの状態で「次へ」「前へ」を押しても何も起きない

**ツリー連動**:
- マッチノードに移動すると右ペインにキー定義が表示される
- ツリー再構築後に検索結果がクリアされる
- ツリー再構築後に検索テキストが残っている場合、再検索が実行される

**UI 状態**:
- マッチ件数ラベルの表示フォーマットが正しい
- 検索バーの UI 要素が正しく配置されている

### テストに必要なフィクスチャ

既存の `wildtree_main_window_with_data` フィクスチャが利用可能。`simple_cards_dir` は `greeting` と `farewell` の 2 キーしかないため、テストケースによっては `multi_file_cards_dir` を使った方がマッチ件数のテストがしやすい。必要に応じて `multi_file_cards_dir` を使うウィンドウフィクスチャを conftest に追加する。
