## 得た知見

### 1. 構造化入力 + 単純アルゴリズムの組み合わせが一発パスの鍵

S3 振り返りで「入力が構造化されている関数は一発パスしやすい」と記録したが、S4 はこれを更に裏付けた。S4 の入力は KeyRegistry（辞書）であり、アルゴリズムは集合差分（`all_keys - referenced_keys`）のみ。4重ループの走査も `set.add()` の連続に過ぎず、バグの入る余地がほぼなかった。

S2（文字列パース）で 2件のバグが発生し、S3（辞書操作）と S4（集合操作）でゼロだった事実は、バグの発生率が「ロジックの複雑さ」ではなく「入力データの構造化度合い」に強く相関することを示している。

### 2. 設計書のエッジケーステーブル + スタブの組み合わせが3サイクル連続で有効

S2 の振り返りで「エッジケースを列挙するだけでなく、各処理ステップがどう振る舞うかを確認する」と提案し、S3 で実践して効果を確認した。S4 でもエッジケーステーブル（12パターン）を設計書に明記し、テスターがそのまま34テストに変換した。

設計段階でのエッジケース明示 → テスト → 実装の流れが確立されたと言える。

### 3. FullPathIndex 不要の判断が正しかった

設計段階で「WildcardRef.name プロパティがフルパス参照も短縮形参照も同じように扱えるため、FullPathIndex は不要」と判断した。この判断により:
- 依存が `core/models.py` のみに限定された（resolver に依存しない）
- 実装がシンプルになった
- テストフィクスチャも FullPathIndex の構築が不要

設計段階で「この機能に本当に必要な入力は何か」を精査したことが功を奏した。

### 4. コメント行の参照カウント方針が設計段階で確定していたのでテスト・実装に迷いがなかった

「コメント行内の参照もカウントするか」は設計上の判断ポイントだったが、設計書で理由（「コメントアウトは一時的な無効化であり、構造的には参照が存在する」）を明記し、却下案テーブルにも含めたことで、テスト・実装フェーズで判断のブレがなかった。

### 5. S3→S4 の連続性がスムーズだった

S3 の振り返りで「S4 では動的参照の内部参照の扱いとコメント行の参照カウントに注意」と具体的に記録していた。S4 の設計段階でこの2点が最初に検討され、設計書の冒頭で方針を確定できた。振り返りの「次に取り組むべき機能」セクションの注意事項が、次サイクルの設計をブートストラップする効果がある。

## 設計と実装のギャップ

**なし。** 設計がそのまま実装に反映された。

3サイクル連続（S2 以降を含めると）でギャップの縮小傾向が続いている:
- S2: ギャップ 2件（空行処理、カンマ分割の深度考慮）
- S3: ギャップ 0件
- S4: ギャップ 0件

ギャップがゼロの主因は「入力が構造化されている」ことと「アルゴリズムが単純」であること。S5（ツリー構築）は再帰的な構造を扱うため、ギャップが発生する可能性がある。循環参照検出、遅延展開の粒度など、設計段階で曖昧さを排除しきれない要素が増える。

## 改善点

### 今後のサイクルへの適用

1. **S5 では再帰的なロジックが入るため、S2 レベルの注意が必要**: S3/S4 の「構造化入力 + 単純アルゴリズム」パターンとは異なり、S5 はツリーの再帰展開を含む。設計書では「処理ステップ x エッジケース」マトリクスを意識し、特に循環参照の検出タイミングと visited セットのスコープ管理を明確にする

2. **設計書のエッジケーステーブルを引き続き充実させる**: 3サイクル連続で効果が実証された。S5 では特に:
   - 循環参照のバリエーション（自己参照、相互参照、3ノード以上の循環）
   - 未解決参照とリテラルの混在
   - 深いネスト（再帰の深さ制限）

3. **「この機能に本当に必要な入力は何か」を設計の初期段階で精査する**: S4 で FullPathIndex を除外した判断のように、不要な依存を設計段階で除外する習慣を維持する

## 設計方針の修正要否

- 現時点では修正不要
- S5（ツリー構築）は S4 の出力（`find_top_trees()` → `list[TopTreeInfo]`）をルートノードとして使用する。TopTreeInfo のフィールド（name, key_def, file_path）が S5 の入力として十分か確認が必要
- S5 では resolver を使うため、S3 のインターフェース（resolve, ResolveResult）との連携も設計時に検証する

## 次に取り組むべき機能

**S5: ツリー構築** (`core/tree_builder.py`)

理由:
- S4（トップツリー検出）の直接の利用者。find_top_trees() が返すトップツリーをルートノードとして展開する
- S6（GUI ツリー表示）の前提。TreeNode ツリーがなければ GUI に表示するデータがない
- v2 の核心機能。参照の再帰展開、循環参照検出、リテラルノード表示など、これまでの S1-S4 の全成果物を統合する

S5 では以下の点に特に注意:
- **循環参照の検出と表示**: visited セットのスコープ管理。パスごとにリセットするか、グローバルに保持するか
- **再帰の深さ制限**: 実データ（43,000キー）での最大深度を事前調査し、安全な上限を設定する
- **リテラルノードの扱い**: ValueEntry のリテラル部分をどう TreeNode に変換するか
- **コメントノードの扱い**: is_commented=True の ValueEntry をどう表示するか
- **遅延展開の要否**: 初期実装では全展開にするか、パフォーマンス上遅延展開が必要か
- **NodeType の使い分け**: REF, LITERAL, DYNAMIC, UNRESOLVED, CIRCULAR, EMPTY の各タイプの判定条件
