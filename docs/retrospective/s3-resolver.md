## 得た知見

### 1. v1 クラスベースから v2 関数ベースへの変更が設計・テスト・実装すべてを簡素化した

v1 の `WildcardResolver` クラスは `_key_registry` と `_cards_dir` をインスタンス変数として保持し、テスト時にもインスタンス化が必要だった。v2 では:

- 設計: 関数シグネチャの確定だけで済み、クラスのライフサイクル設計が不要
- テスト: レジストリとインデックスを引数で渡すだけ。インスタンス化やモックが不要
- 実装: 各関数が独立しており、関数単体で正しさを保証できる

状態を持たないデータ変換パイプラインでは、クラスより関数ベースの方が適切。ただしこれは resolver のように「入力を受け取って結果を返すだけ」の責務に限った話であり、ツリー構築（S5）のように段階的に構造を組み立てる場合はクラスが適切な可能性がある。

### 2. フルパスインデックスの事前構築方式が解決ロジックを極めて単純にした

v1 では解決時に `cards_dir` からの相対パスを動的に計算していた。v2 では `build_full_path_index()` で事前に辞書を構築し、`resolve()` は辞書引き 2回（フルパス → 短縮形）だけの単純なロジックになった。

この結果:
- `resolve()` の実装が 40 行未満で完了
- 一発でテストパスした主因。ロジックが単純すぎてバグの入る余地がなかった
- パフォーマンスも O(1) ルックアップで問題なし

### 3. 設計書のエッジケース表が実装の見落とし防止に直結した

設計書に以下のエッジケースを明記していたことで、実装時に自然にカバーできた:
- 空文字列の `full_path`
- `"cards/"` のみの `full_path`（キー名部分がない）
- `"cards/"` プレフィックスの除去後が空文字列
- フルパス解決では `is_ambiguous` が常に `False`

S2 の教訓「処理ステップ x エッジケースのマトリクス」を意識した結果、今回はエッジケース起因のバグがゼロだった。

### 4. conftest.py のフィクスチャ設計が「目的別分離」原則に沿っていた

S3 で新規追加した 3 フィクスチャ:
- `sample_registry`: resolve のユニットテスト用
- `sample_full_path_index`: resolve のユニットテスト用（sample_registry に依存）
- `duplicate_key_registry`: find_duplicate_keys のテスト用

これらは目的が明確に異なるため、S2 振り返りの「共有フィクスチャに複数の目的を兼ねさせない」ナレッジに従い、別フィクスチャとして分離した。結果として、テスト変更時の影響範囲が最小化されている。

### 5. 循環参照検出を resolver から分離した判断が正しかった

循環参照は「ツリー構築時の再帰的な参照辿り」で発生するものであり、resolver は「1つの参照名 → 1つの KeyDefinition」の単発マッピングに徹するべきという設計判断が正しかった。これにより:
- resolver の責務が明確になり、テストもシンプルになった
- resolver が「文脈」（どの経路から到達したか）を知る必要がなくなった
- S5（tree_builder）で循環参照を適切に処理できる設計の自由度が保たれた

### 6. レビュー指摘の Literal 型化は型安全性を向上させた

`ResolveResult.method` を `str` から `Literal["full_path", "short"]` に変更したことで:
- タイポが型チェッカーで検出可能になった
- 将来的に解決方法が増えた場合に、全箇所で対応漏れを検出できる
- 実装コストはほぼゼロ（import 追加 + 型アノテーション変更のみ）

## 設計と実装のギャップ

**なし。** 設計がそのまま実装に反映された。

これは S1（データモデル）とは異なる理由で達成された。S1 は「設計 = コード」の性質上ギャップが生じにくかった。S3 は実装ロジックがあるが、フルパスインデックスの事前構築方式が解決ロジックを極めて単純にしたため、設計段階で予見したロジックと実装ロジックが完全に一致した。

S2 で発生した 2 件のギャップ（空行処理、カンマ分割の深度考慮）のような「暗黙の前提が壊れる」パターンは、S3 では発生しなかった。理由として:
- 文字列パースロジックがない（S2 はテキスト処理、S3 は辞書操作）
- 行ベース処理がない（S2 の暗黙の前提は行の分類ロジックに起因していた）
- 入力データが既に構造化されている（KeyRegistry, FullPathIndex）

## 改善点

### 今後のサイクルへの適用

1. **「入力が構造化されている関数」は一発パスしやすい**: S3 の入力は KeyRegistry（辞書）と FullPathIndex（辞書）であり、文字列パースのような曖昧さがなかった。S4（トップツリー検出）と S5（ツリー構築）も KeyRegistry を入力とするため、同様のアプローチが有効

2. **レビュー指摘の Literal 型化を S4 以降にも適用**: Enum 的な文字列フィールドには Literal 型を使う。設計段階で決められる

3. **レビュー指摘「同名キーが同一ファイル内に複数」のテスト追加**: S3 のレビューで推奨されたが、テストファイル変更禁止でスキップした。次のサイクルのテストフェーズで対応を検討する

## 設計方針の修正要否

- 現時点では修正不要
- S4（トップツリー検出）は S3 の出力（FullPathIndex, resolve 関数）を使うため、S3 のインターフェースが S4 の入力として十分かを S4 設計時に検証する
- S5（ツリー構築）でも resolve 関数を使うが、循環参照検出は S5 側で実装する方針を維持

## 次に取り組むべき機能

**S4: トップツリー検出** (`core/top_tree.py`)

理由:
- S3（名前解決）の直接の利用者。resolve を使って「どのキーからも参照されていないキー = エントリポイント」を特定する
- S5（ツリー構築）の前提。ツリーのルートとなるキーを特定する必要がある
- 比較的小規模な機能であり、S5 の前に完了させることで S5 の設計に集中できる

S4 では以下の点に特に注意:
- 「参照されていないキー」の定義を明確にする（動的参照の内部参照も考慮するか）
- コメントアウトされた参照行の扱い（コメントアウトされた行からの参照はカウントするか）
- 42,957 キーでのパフォーマンス（全キーを走査するが、O(n) で十分）
