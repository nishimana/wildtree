# 振り返り: W2 壊れた参照のハイライト

日付: 2026-03-18

## サイクル概要

- **プロジェクト**: WildTree -- Dynamic Prompts ワイルドカード YAML のツリービューア
- **技術スタック**: Python 3.12 + PySide6
- **タスク種別**: 混合（`core/wildcard_parser.py` と `gui/main_window.py` の両方を変更）
- **スコープ**: `TreeNode` に `is_unresolved` フィールド追加、`_build_node()` で未解決参照をツリーに含める、GUI で赤字表示
- **成果**: 全110テストパス（既存90 + 新規18 + 既存1更新）、レビュー OK 判定（重要指摘なし、推奨2件は修正済み）

## 開発の流れ

1. **設計**: architect が設計書作成 + `TreeNode` に `is_unresolved` フィールドのスタブ追加 + GUI に定数・import のスタブ追加
2. **テスト**: tester が 18 テスト作成（core 12 + GUI 5 + 定数 3）+ conftest に 4 フィクスチャ追加 + 既存テスト1件更新
3. **実装**: implementer が `_build_node()` と `_populate_tree_item()` / `_build_and_display_tree()` を変更。全110テストパス
4. **レビュー**: OK 判定。推奨2件（docstring 不整合、防御コードのコメント不足）を修正
5. **修正**: 推奨2件を修正。全110テストパス

## うまくいったこと

### core と GUI の責務分離が設計通りに機能した

- `is_unresolved` フラグによる core → GUI の情報伝達設計が機能した。GUI 側は `is_unresolved` フラグを見るだけで赤字表示を判定でき、名前解決ロジックを持たない
- 代替案（GUI 側で `resolve()` を呼んで壊れた参照を判定する案）を設計段階で却下したことで、関心の分離が維持された

### 既存テストへの影響を設計段階で特定できた

- 設計書で `test_ツリー構築_未解決参照はツリーに含まれない` (test_parser.py) が W2 で失敗することを明記した
- tester がこのテストを W2 の新しい振る舞いに合わせて更新し、混乱なく移行できた
- 設計段階での既存テスト影響分析が、テスト作成の効率を上げた

### W1 で確立したパターンの再利用

- `is_circular` と同じパターン（`_populate_tree_item()` でのフラグチェック + 表示変更）を `is_unresolved` にも適用
- 既存パターンに沿うことで、設計・実装・レビューの全段階でコストが低下した

### 設計書の変更指示が具体的だった

- 6箇所の変更を「変更前→変更後」のコード差分レベルで記述
- implementer への指示が明確で、解釈の余地がなかった
- 結果としてレビューの「重要」指摘はゼロだった

### conftest フィクスチャの設計が効果的だった

- `broken_ref_cards_dir`、`broken_ref_fullpath_cards_dir`、`all_broken_ref_cards_dir`、`broken_and_circular_cards_dir` の 4 フィクスチャを追加
- 各フィクスチャが 1 つのテストシナリオに対応し、テストコードの可読性が高い
- `yaml_factory` と組み合わせた柔軟なテストデータ生成も一部で活用

## 設計と実装のギャップ

### docstring の古い記述が残った（レビュー指摘 #1）

- `build_tree()` の docstring に W2 以前の「未解決参照はスキップ」という記述が残っていた
- 設計書の「implementer への変更指示」にはコード変更のみ記載し、docstring の更新を明記していなかった
- implementer はコード変更指示に忠実に従ったが、関連する docstring の更新は行わなかった
- **教訓**: ロジック変更を伴うコード変更では、関連する docstring の更新も設計書の変更指示に含めるべき

### 防御コードのコメント不足（レビュー指摘 #2）

- `_build_and_display_tree()` でルートノードの `is_unresolved` チェックが防御的コードであることを設計書に記載していたが、コード上のコメントとして指示しなかった
- 設計書に「防御的にチェック」と書いてあっても、実装時にそのコンテキストがコードコメントとして転写されるとは限らない
- **教訓**: 到達困難なコードパスには、設計書の変更指示にコメント文言も含める

## 次回以降に活かすべき改善点

### 1. ロジック変更時の docstring 更新を変更指示に含める

今回、コードの振る舞いが変わったにもかかわらず docstring が古いままだった。設計書の「implementer への変更指示」セクションでは、コード差分だけでなく、関連する docstring の更新も明示的に指示すべき。特に関数の docstring に「この関数は X する / X しない」という記述がある場合、振る舞い変更時には必ず確認が必要。

### 2. 防御的コード・到達困難コードのコメント方針

「設計上は到達しないが防御的にチェックする」コードには、その旨をコードコメントとして残すことを変更指示に含める。設計書の本文に理由が書いてあっても、実装者がそれをコードコメントに転記するとは限らない。

### 3. resolve() 二重呼び出しの技術的負債

レビュー提案 #4 で指摘された `resolve()` の二重呼び出し（正常参照の場合、`_build_node()` 内で1回、子ノード構築ループで1回）は、現時点ではパフォーマンス影響がないためスキップしたが、ツリーが大きくなった場合に問題になる可能性がある。将来の最適化候補として記録する。
