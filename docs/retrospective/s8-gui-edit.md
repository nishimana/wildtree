## 得た知見

### 1. 8サイクル連続で初回テスト全パス（S1 除く）

S2-S8 の7サイクルで「設計 → テスト → 実装」の流れが安定して機能し続けている。S7 の1件の失敗はテストヘルパーの問題であり、今回の S8 では初回テストが完全に全パスした。設計書にインターフェースと振る舞いを明確に定義するプラクティスが定着している。

### 2. 設計書にないガード追加を実装者が正しく判断

`_on_top_tree_selected` にも `_is_populating` ガードを追加する必要があったが、設計書には `_on_item_changed` のガードしか記載されていなかった。実装者が populate_model を呼ぶ全パスを確認し、自主的にガードを追加した。これは「同一概念の複数ハンドラには同じガードパターンを適用する」原則が実装段階で正しく機能した例。

ただし、設計段階でこのパターンを検出すべきだった。設計書の「_is_populating ガードが必要な理由」セクションでは `populate_model` 中のシグナル発火問題を議論したが、`populate_model` を呼ぶ既存メソッド（`_on_top_tree_selected`）への波及を明示的に記載しなかった。

### 3. エラーパスの状態復元は設計段階で列挙すべき

toggle_comment が失敗した場合、チェックボックスの状態は既に変更済み（Qt の itemChanged はチェック状態変更後に発火する）だが、ファイルは変更されていない。この不整合を解消するために `_rebuild_tree()` で正しい状態に戻す必要があった。

設計書の「エラーハンドリング」セクションでは「QMessageBox.warning でエラーメッセージを表示。ツリーを変更しない」としていたが、「チェックボックスの状態は既に変更されている」ことを見落としていた。正しくは「ツリーを再構築して正しい状態に戻す」べきだった。

### 4. core/ の変更なしで GUI 機能を実現

S6（GUI 表示）+ S7（コア層エディタ）の設計が適切だったため、S8 では core/ を一切変更せずに GUI 層のみの変更で編集機能を実現できた。パイプライン（toggle_comment → refresh_registry → build_full_path_index → build_tree → populate_model）をそのまま再利用する「全ツリー再構築方式」が正しく機能した。

## 改善点

### 今後のサイクルへの適用

1. **既存メソッドへの波及を設計段階で列挙する**: 新しいガードやフラグを導入する場合、そのフラグが必要な既存メソッドを全て列挙する。今回の `_is_populating` のケースでは、設計書に「populate_model を呼ぶ既存メソッド一覧」を含めるべきだった

2. **エラーパスでの UI 状態の正確な記述**: Qt のシグナルは状態変更後に発火するため、itemChanged ハンドラに入った時点でチェックボックスの状態は既に変わっている。設計書のエラーハンドリングセクションでは「この時点の UI 状態」を正確に記述し、どのような復元操作が必要かを設計する
