## 得た知見

### 1. GUI モジュールでもコア層と同じ「設計 → テスト → 実装」の流れが有効

S6 は v2 で初めて PySide6 が登場するモジュールだが、コア層（S2-S5）で確立した開発サイクルがそのまま機能した。GUI 固有のテスト課題（QApplication のセッションスコープフィクスチャ、offscreen プラットフォーム）はあったが、テスト可能な設計（関数ベースの tree_model, detail_pane）により、コア層と同じ品質を維持できた。

### 2. GUI 層の責務分離（3モジュール分割）がテスト容易性を確保した

- `tree_model.py`: TreeNode → QStandardItem のマッピングロジックに集中。Qt モデル API でテスト可能
- `detail_pane.py`: Qt ウィジェットに依存しない純粋関数（入力: TreeNode → 出力: str）。QApplication 不要でテスト可能
- `app.py`: 統合・シグナル接続のみ。QApplication が必要なテストはここに集約

detail_pane.py を Qt 非依存にしたことで、41テストが QApplication なしで実行可能になった。これは `principles.md` の「GUI非依存の純粋関数からテストを始める」の実践例。

### 3. 設計書の「LITERAL フォーマット」に設計-実装ギャップが発生

設計書の LITERAL 詳細フォーマットでは「所属キー情報は表示しない」と記載していたが、テスト作成時にテスターがこの意図を正しく反映した。一方で、レビュー時に設計書の表示フォーマット例と実装の出力形式にズレがあることが検出された。修正は設計書側を実装に合わせる形で対応した。

このギャップは「設計書のフォーマット例が具体的すぎて、実装時に自然に改善された部分」に起因する。フォーマット例は「構造」を示すに留め、「文言の細部」は実装時に確定する方針が妥当。

### 4. 5サイクル連続で初回テスト全パス

S2-S6 の5サイクル連続で初回テスト全パス。S6 はコア層とは異なるドメイン（PySide6 GUI）だが、設計書のインターフェース定義 + 詳細な表示仕様テーブル + フォーマット例がテスト・実装の精度を維持した。

### 5. コア層パイプラインの統合が GUI 層で検証された

app.py の `_load_cards_dir()` メソッドで S1-S5 の全モジュールを連結するパイプライン（scan → parse → resolve → find_top_trees → build_tree → populate_model）が統合テストで検証された。これは S1-S5 の各モジュール間の互換性を GUI 層から確認する結合テストとして機能した。

## 改善点

### 今後のサイクルへの適用

1. **設計書のフォーマット例は「構造」を示し、「文言の細部」は実装で確定する方針を明確にする**: S6 で設計書のフォーマット例の文言と実装出力のズレが発生した。設計書のフォーマット例では構造（キー名、ファイルパス、行番号、値行一覧のレイアウト）を明示し、具体的な文言は「実装で確定」とする

2. **テストヘルパー（`_build_tree_from_cards_dir` 等）の conftest.py 集約**: S5 振り返りから継続。3ファイル（test_tree_builder.py, test_tree_model.py, test_gui_app.py）で同一パターンのヘルパーが重複している。次回テスト作成時に conftest.py への集約を検討する

3. **S7（YAML エディタ）ではファイル書き込みのエッジケースに注意**: コメント切替は行単位の書き換えであり、YAML パーサーを使わない。行ベースの書き換えではインデント・改行コード（CRLF/LF）の保持が課題になる

## 設計方針の修正要否

- 現時点では修正不要
- S6 の開発サイクルは、コア層（S2-S5）と同じパターンで安定して動作した
- GUI 層の責務分離（統合 / モデル変換 / 詳細表示）の方針は S8（GUI 編集機能）でも継続可能
- S8 ではチェックボックスの表示・操作が加わるため、カスタムデリゲートの導入を検討する（S6 設計判断 #5 で示唆済み）

## 次に取り組むべき機能

**推奨: S7（YAML エディタ `core/editor.py`）**

理由:
- S8（GUI 編集機能）の前提モジュール。S8 は S6 + S7 に依存する
- S7 はコア層のモジュールであり、S2-S5 と同じ開発パターンが適用可能（テスト容易性が高い）
- S9（動的参照の展開表示）や S10（キー名参照自動解決）は読み取り専用機能の拡張であり、S7→S8 の編集機能パスが優先度が高い（v2 の核心機能「その場で編集」の実現に直結する）

S7 設計時の注意点:
- YAML の行ベース書き換え（PyYAML を使わない。S2 パーサーと同じ方針）
- コメント切替（`# ` プレフィックスの付与/除去）の行単位操作
- 改行コード（CRLF/LF）の保持。Windows 環境での動作を考慮
- ファイルの部分書き換え vs 全体書き換えのトレードオフ
- 書き換え後のレジストリ再構築の設計（差分更新 vs 全再構築）
